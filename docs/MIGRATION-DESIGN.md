# Migration Design: OpenCode → Gewu

- **Version:** 1.0
- **Date:** 2026-07-09

## Overview

Migrate data from OpenCode (existing collaboration platform) to Gewu (new AI-driven platform). Migration handles approximately 25,000 users, 100,000 sessions, and 50,000 agents.

## Scope

### In Scope

| Source | Target |
|--------|--------|
| `opencode.users` | `gewu.users` |
| `opencode.sessions` | `gewu.project` + `gewu.session` |
| `opencode.message` + `opencode.message_part` | `gewu.session_message` |
| `opencode.agents` | `gewu.agent` |
| `opencode.tools` | `gewu.agent_tool` |

### Out of Scope

- **Credentials** — Re-auth required per security policy
- **Audit logs** — New format incompatible
- **File attachments** — Re-upload required
- **API keys** — Re-generate required

## ID Mapping Strategy

OpenCode IDs are already CHAR(26) compatible with ULID format. No transformation needed for existing IDs. New IDs generated by Gewu will use ULID (26-char Crockford Base32). SM3 hash used for reconciliation mapping.

## V1 Message Merge

V1 platform stored messages across `message` + `message_part` tables (1:N). V2 uses `session_message` with JSON `content` field. Merge strategy:

1. Group V1 `message_part` records by parent `message_id`
2. Concatenate in order with separator
3. Insert as single `session_message` record
4. Attach all original metadata (timestamps, author info)

## Migration Process

### Pre-migration

1. Full database backup (both source and target)
2. Run data quality check on source
3. Generate validation report
4. Disable writes to source system

### Migration

1. Migrate users (base entity)
2. Migrate sessions (depends on users)
3. Migrate messages (depends on sessions)
4. Migrate agents (depends on users)
5. Reconcile ULID mappings

### Post-migration

1. Validate record counts
2. Verify foreign key integrity
3. Run data integrity checks
4. Generate final validation report
5. Enable application and verify user login

## Rollback Plan

Four-tier rollback:

| Tier | Trigger | Action |
|------|---------|--------|
| 1 — Full restore | Any failure | Restore from backup |
| 2 — Partial rollback | Data quality issues | Roll back by table |
| 3 — Checkpoint rollback | Partial migration failures | Roll back to last checkpoint |
| 4 — Application rollback | Post-migration issues | Switch to old deployment |

## Risk Assessment

| Risk | Impact | Probability | Mitigation |
|------|--------|------------|------------|
| Data loss | High | Low | Full backup before migration |
| ID collision | Medium | Low | ULID recomputation with collision detection |
| FK violation | High | Medium | Migrate in dependency order |
| Downtime > 4h | Medium | Low | Rollback checkpoint every 10K records |
